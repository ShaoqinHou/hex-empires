## Summary

Phase 1.5.1 wires a CSS grid to `.game-app` and removes the hard-coded `h-12`
Tailwind class from TopBar so chrome bar heights are driven by layout-tokens.css.
The commit is clean on all engine rules, introduces no raw hex, and no panel/HUD
anti-patterns are present. One structural concern: the grid targets `.flex-1` as a
child selector, coupling grid layout to a Tailwind utility class name.

## Findings

### F-67636928
- severity: warn
- file: packages/web/src/index.css
- line: 200
- rule: tech-conventions.md § "TypeScript"
- offender: `.game-app > .flex-1 { grid-area: canvas; }`
- message: The canvas grid-area assignment targets `.flex-1`, a Tailwind utility
  class — if the canvas wrapper div ever loses or changes that class the grid
  layout silently breaks with no TypeScript error or failing test.
- suggested-fix: Add a semantic class `layout-canvas-area` to the canvas wrapper
  in App.tsx/GameUI, and target `.layout-canvas-area` in CSS instead of `.flex-1`.
- state: open

### F-fff5b744
- severity: note
- file: packages/web/e2e/viewport-grid.spec.ts
- line: 19
- rule: testing.md § "Assertion Rules"
- offender: `{ topbarHeight: 56, bottombarHeight: 48 }, ...`
- message: Expected pixel heights duplicate layout-tokens.css values inline; if the

- suggested-fix: Add inline comments cross-referencing the token source
  (e.g. `// --chrome-topbar-height standard`) so maintainers know where these
  numbers originate.
- state: open
